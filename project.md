  # **企微-微信私域逆向项目**

  ## **一、项目背景** 

  随着业务运营场景的不断拓展和精细化需求的提升，现有官方接口能力无法满足高效运营及服务需求，核心痛点主要体现在两个维度：

  ​            1.     企业微信官方API能力不足：官方服务API在群管理和消息管理核心场景（如消息群发、群批量创建）存在诸多限制，且其功能仅为PC端企业微信的子集，大量PC端原生能力未对外开放，无法支撑运营效率提升。

  ​            2.     微信无官方开放接口支撑：微信未提供任何开放API，无法满足超核用户一对一精细化服务、高效社交运营等核心业务需求，成为业务落地的关键瓶颈。

  针对上述核心痛点，我们通过对PC端微信及PC端企业微信的汇编代码进行逆向研究、解析其底层调用逻辑，通过植入自定义调用代码的方式，突破官方接口限制，从而实现对PC端微信、PC端企业微信的自动化控制，覆盖官方接口未开放的能力，支撑业务高效运营。

   

  ## **二****、****项目整体层级架构**

  ​                ● 业务层：主要为私域项目的服务对象，包括超核、知己、心悦等；直接面向游戏用户，提供业务功能和服务；

  ​                ● 操作层：主要为运营开发人员提供账号操作和管理工具，包括TITAN管理端、UPBOT管理端、AIPRO管理端和IMT_WEB。TITAN客户端面向TITAN业务人员，AIPRO面向知己业务人员，UPBOT管理端用于知己、心悦、王者赛宝、元梦之星等业务的管理。而IMT_WEB目前为小组内部使用，提供各类业务账号的监控、操作管理以及机器状态监测；

  ​                ● 服务层：主要对操作层和逻辑层间提供事件流通服务、链接管理等，ww_node_server对接知几、titan后台与ww_robot_proxy，wx_robot_proxy直接对接心悦后台和titan后台，imt_node_server对接imt_web与ww/wx_robot_proxy。

  ​                ● 逻辑层：主要负责一对一控制客户端（如WW/WX客户端）的代理机器人（proxy-robot）管理，涉及事件下发的逻辑处理，以及微信企微客户端状态和实例状态的信息上报。

  ​                ● 逆向层：专注于对微信企微客户端的逆向工程以提供必要的底层支持，包括hook点寻找、hook逻辑、代码注入以及反反外挂；

  ​                ● 工具层：包括最终运行在机器上的被控对象，如微信、企微和QQ客户端，这些都是逆向层和逻辑层操作的目标；

  小组内的私域工程工作主要集中于imt_node_server、imt-web、ww-node-server、ww-robot-proxy、wx-robot-proxy；

  ### **2****.****1** **ww/wx-robot-proxy（PC企微/微信的「远程遥控器」）**

  本质是在业务后台或企微server和PC企微/微信之间做了一层桥接：后台只发指令，本地代理负责打通与微信进程的通道，并完成实际操作和结果上报。

  具体来说：后台（例如 Titan后台）要远程控制某台机器上的 PC 微信做：扫码登录、发消息、加好友、建群拉人、朋友圈等。微信本身没有对外开放的 API，所以需要一个本地代理：proxy_controller主进程负责与后台通信、解析指令，再通过控制robot注入dll到微信进程去执行真实操作，最后把执行结果和微信内发生的事件（如收到消息、好友申请）回传给后台。

  ​                ● robot_proxy_controller: 负责拉起proxy，创建和控制robot控制器；

  ​                ● robotManager：负责管理某一robot与连接；

  ​                ● robot：负责某一微信/企微账号的控制，包括登陆、发消息等具体功能；

  ​                ● wx_inject_helper.dll：将控制代码注入到微信/企微进程，执行真实操作；

  ​                ● wx_connection：连接业务后台，收发事件；

  ​                ● monitor_connection：连接imt后台和监控后台，上报与运维；

  ![img](https://wdoc-76491.picgzc.qpic.cn/MTY4ODg1NzMzNTU5NTQ2Mw_260152_-pIszg5OlhZX9TfD_1772443766?w=1141&h=710&type=image/png)

  控制微信/企微核心原理：即DLL远程注入，利用编写的robot_proxy.exe将Helper.dll注入到微信进程中，使得微信进程成为傀儡进程，响应下发的各种指令，并上报微信进程中的各项数据。

  ![img](https://wdoc-76491.picgzc.qpic.cn/MTY4ODg1NzMzNTU5NTQ2Mw_507598_GrlNKfoF_eWGJHkC_1772443640?w=686&h=742&type=image/png)

  ### **2****.****2** **ww-node-server（业务后台与ww-robot-proxy之间的桥梁）**

  ![img](https://wdoc-76491.picgzc.qpic.cn/MTY4ODg1NzMzNTU5NTQ2Mw_539694_q0Yu_MHXmLOMXcLY_1772443608?w=1783&h=844&type=image/png)

  整个系统采用微服务架构，由 10 个独立服务组成，分为三个层次：

  ​                ● 接入层：由 Gateway 网关服务承担，基于 Koa 框架的 HTTP 服务，负责接收业务方的所有 API 请求；

  ​                ● 业务层：包含了五个核心服务。

  ​                ○ Login 服务负责机器人的登录认证和状态管理。

  ​                ○ User 服务管理企业内部员工和外部联系人的信息，包括好友关系的建立、通讯录的同步等。

  ​                ○ Group 服务负责群组的创建、成员变更、群信息同步等功能。

  ​                ○ Message 服务是消息处理的核心，支持文本、图片、视频、语音、文件、小程序、链接等多种消息类型的收发，同时负责将收到的消息回调给业务方。

  ​                ○ User-Queue 服务是一个基于 Redis 的优先级任务队列，用于消息发送的削峰填谷和频率控制，避免因发送过快触发企业微信的风控。

  ​                ● 连接层：由 Socket-Manager 和 Socket-Srv 两个服务组成。

  ​                ○ Socket-Srv 服务维护与 企微控制器的 WebSocket 长连接，每个企微账号登录后会与服务端建立一条持久连接，用于双向通信。

  ​                ○ Socket-Manager 服务则负责连接的调度和管理，它维护着所有在线连接的状态，当业务需要向某个机器人发送消息时，它会找到该机器人对应的连接，并将任务分发到正确的 Socket-Srv 实例

  ### **2****.****3** **imt-web/imt-server/update_server（面向组内的****imt****私域****工具****管理平台）**

  IMT私域工具管理平台是一套面向组内管理系统，主要用于管理和运维 QQ、微信、企业微信 三种平台的机器人实例（Proxy）。

  具体结构：

  ​                ● 底层是运行在多台 Windows 服务器上的 Proxy 实例（ww_robot_proxy 和 wx_robot_proxy），它们直接与真实的 IM 客户端交互。每个 Proxy 启动后会连接到 imt-server，注册自己的存在并开始定期上报状态数据。

  ​                ● imt-server 作为中枢，维护着所有 Proxy 的实时状态，存储配置信息和历史数据，并对外提供 API 接口。

  ​                ● imt-web 通过调用 imt-server 的 API，将这些数据以可视化的方式展示给运维人员，同时也将用户的操作指令传递给后端。

  ​                ● 当 IM 客户端发布新版本时，在 imt-web 创建升级任务，然后使用 update_server 工具完成 Hook 迁移，编译出新版本的 Proxy，最后通过 imt-web 的机器管理功能将新版本发布到各台服务器上。
