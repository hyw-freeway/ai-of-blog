## 一、先讲一个故事

想象你开了一家**餐厅**：

```
                    🏠 你的餐厅
                         │
    ┌────────────────────┼────────────────────┐
    │                    │                    │
    │   👨‍🍳 厨师 (后端)    │   📋 菜单 (前端)    │
    │   做菜、处理订单    │   给客人看的静态内容 │
    │                    │                    │
    └────────────────────┴────────────────────┘
```

现在问题来了：

1. **客人来了谁接待？** —— 需要一个**前台接待员**
2. **厨师生病了怎么办？** —— 需要一个**厨师长**盯着

在网站世界里：
- **Nginx** = 前台接待员
- **PM2** = 厨师长

## 二、Nginx 是什么

### 2.1 一句话解释

> **Nginx 就是餐厅的前台接待员**，负责接待所有客人，然后把他们引导到正确的地方。

### 2.2 生活中的比喻

想象一个大型商场的**服务台**：

```
                        🛎️ 商场服务台 (Nginx)
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
        ▼                     ▼                     ▼
   "我要看电影"          "我要吃饭"           "我要买衣服"
        │                     │                     │
        ▼                     ▼                     ▼
   🎬 电影院              🍔 美食城            👔 服装店
   (前端静态文件)        (后端 API)          (其他服务)
```

服务台小姐姐不会自己放电影、做饭、卖衣服，但她知道：
- 要看电影 → 往左走到电影院
- 要吃饭 → 直走到美食城
- 要买衣服 → 往右走到服装店

**Nginx 也是一样**：它不处理具体业务，只负责把请求**分发**到正确的地方。

### 2.3 Nginx 做的三件事

#### 事情一：提供静态文件（当图书管理员）

```
用户请求: "我要看首页"
         │
         ▼
    ┌─────────┐
    │  Nginx  │ ──▶ 去 /var/www/dist 文件夹找 index.html
    └─────────┘
         │
         ▼
    📄 返回 index.html 给用户
```

这就像图书管理员，你说要借《三国演义》，他就去书架上找到这本书给你。

前端打包后的文件（HTML、CSS、JS、图片）都是**静态文件**，Nginx 可以直接返回，速度非常快。

#### 事情二：反向代理（当传话筒）

```
用户请求: "我要获取文章列表" (访问 /api/articles)
         │
         ▼
    ┌─────────┐
    │  Nginx  │ ──▶ 这个请求我处理不了，转给后端 Node.js
    └─────────┘
         │
         ▼
    ┌─────────────┐
    │  Node.js    │ ──▶ 查询数据库，处理业务逻辑
    │  (后端)     │
    └─────────────┘
         │
         ▼
    📦 返回 JSON 数据给 Nginx，再由 Nginx 返回给用户
```

**什么是"反向代理"？**

- **正向代理**：你让朋友帮你买奶茶（你知道奶茶店在哪，但让别人代劳）
- **反向代理**：你跟服务台说要喝奶茶，服务台帮你联系奶茶店（你不需要知道奶茶店在哪）

Nginx 做的就是反向代理：用户只需要访问一个地址，Nginx 会自动把请求转发到正确的后端服务。

#### 事情三：负载均衡（当派单员）

如果你的餐厅生意太好，一个厨师忙不过来，需要请多个厨师：

```
                    ┌─────────┐
                    │  Nginx  │
                    └────┬────┘
                         │
         ┌───────────────┼───────────────┐
         │               │               │
         ▼               ▼               ▼
    ┌─────────┐    ┌─────────┐    ┌─────────┐
    │ 厨师 A  │    │ 厨师 B  │    │ 厨师 C  │
    │ :3001   │    │ :3002   │    │ :3003   │
    └─────────┘    └─────────┘    └─────────┘
```

Nginx 会把订单**均匀分配**给不同的厨师，避免一个人累死、其他人闲着。

> 注：我们的博客项目目前只有一个后端实例，用不到负载均衡。但当网站流量大了，这个功能就很有用。

## 三、PM2 是什么

### 3.1 一句话解释

> **PM2 就是厨师长**，负责监督厨师（Node.js）的工作，确保他不偷懒、不罢工。

### 3.2 为什么需要 PM2？

想象一下没有厨师长会发生什么：

```
😱 问题1：厨师睡着了
   厨师做着做着，累了，睡着了（程序崩溃）
   没人叫醒他，客人就吃不上饭了

😱 问题2：厨师下班了
   下午6点，厨师说"我下班了"就走了（终端关闭）
   晚上来的客人就没人做菜了

😱 问题3：不知道厨师在干嘛
   厨师说他很忙，但你不知道他到底做了多少菜（没有日志）
```

### 3.3 PM2 解决的问题

```
┌──────────────────────────────────────────────────┐
│                    👨‍🍳 厨师长 (PM2)                │
│                                                  │
│  ✅ 监督厨师：厨师睡着了？立刻叫醒他！（自动重启）   │
│  ✅ 24小时值班：你走了，厨师长还在（后台运行）      │
│  ✅ 记录工作：每道菜都记录在案（日志管理）          │
│  ✅ 开店自动上班：餐厅开门，厨师自动到岗（开机自启）│
│                                                  │
└──────────────────────────────────────────────────┘
```

### 3.4 没有 PM2 会怎样？

```bash
# 直接运行 Node.js
node app.js

# 问题1：终端一关，程序就停了
# 问题2：程序报错崩溃，没人重启
# 问题3：不知道程序运行得怎么样
```

有了 PM2：

```bash
# 用 PM2 启动
pm2 start app.js

# 现在：
# ✅ 关闭终端，程序继续运行
# ✅ 程序崩溃，自动重启
# ✅ 随时查看日志和状态
```

## 四、它们如何配合工作

### 4.1 完整的请求流程

当你在浏览器输入 `http://106.55.56.92` 访问博客时，发生了什么？

```
👤 用户浏览器
    │
    │ ① 发起请求: http://106.55.56.92
    ▼
┌─────────────────────────────────────────────────────────────┐
│                        🖥️ 服务器                            │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │                    Nginx (:80)                      │  │
│   │                                                     │  │
│   │  ② 收到请求，分析 URL                                │  │
│   │     │                                               │  │
│   │     ├─▶ URL 是 "/" 或 "/login" 等                   │  │
│   │     │   ③ 返回前端静态文件 (dist/index.html)        │  │
│   │     │                                               │  │
│   │     └─▶ URL 是 "/api/xxx"                           │  │
│   │         ④ 转发给后端处理                             │  │
│   │              │                                      │  │
│   └──────────────┼──────────────────────────────────────┘  │
│                  │                                         │
│                  ▼                                         │
│   ┌─────────────────────────────────────────────────────┐  │
│   │              PM2 管理的 Node.js (:3001)              │  │
│   │                                                     │  │
│   │  ⑤ 处理 API 请求                                    │  │
│   │     - 查询数据库                                    │  │
│   │     - 执行业务逻辑                                  │  │
│   │     - 返回 JSON 数据                                │  │
│   │              │                                      │  │
│   └──────────────┼──────────────────────────────────────┘  │
│                  │                                         │
│                  ▼                                         │
│   ┌─────────────────────────────────────────────────────┐  │
│   │                    MySQL (:3306)                    │  │
│   │                                                     │  │
│   │  ⑥ 存储和查询数据                                   │  │
│   │                                                     │  │
│   └─────────────────────────────────────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
    │
    │ ⑦ 数据原路返回
    ▼
👤 用户浏览器显示页面
```

### 4.2 为什么要这样设计？

| 角色 | 擅长的事 | 不擅长的事 |
|------|----------|------------|
| **Nginx** | 处理静态文件、分发请求、高并发 | 业务逻辑、数据库操作 |
| **Node.js** | 业务逻辑、数据库操作、API | 直接暴露到公网、进程管理 |
| **PM2** | 进程管理、日志、自动重启 | 处理 HTTP 请求 |

**分工协作，各司其职**，就像餐厅里：
- 前台接待（Nginx）不会做菜
- 厨师（Node.js）不会接待客人
- 厨师长（PM2）不会自己上灶

## 五、Nginx 详细讲解

### 5.1 Nginx 配置文件解读

我们的配置文件在 `/etc/nginx/conf.d/blog.conf`，逐行解释：

```nginx
server {
    # 监听 80 端口（HTTP 默认端口）
    # 就像餐厅把门牌号设为 80 号
    listen 80;
    
    # 服务器名称（可以是域名或 IP）
    # 就像餐厅的名字
    server_name 106.55.56.92;

    # ========== 规则1：处理前端请求 ==========
    location / {
        # 静态文件存放目录
        # 就像告诉服务员：菜单放在这个抽屉里
        root /var/www/ai-of-blog/front_end_of_blog/dist;
        
        # 默认首页文件
        index index.html;
        
        # 关键！SPA 单页应用支持
        # 意思是：找不到文件就返回 index.html
        # 这样 /login、/article/1 这些路由才能正常工作
        try_files $uri $uri/ /index.html;
    }

    # ========== 规则2：处理 API 请求 ==========
    location /api {
        # 转发到后端 Node.js 服务
        # 就像说：这个订单交给后厨处理
        proxy_pass http://127.0.0.1:3001;
        
        # 传递客户端真实信息
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        
        # 禁用缓冲，支持流式响应（AI 生成摘要时需要）
        proxy_buffering off;
    }

    # ========== 规则3：处理上传文件访问 ==========
    location /uploads {
        proxy_pass http://127.0.0.1:3001;
    }

    # 限制上传文件大小为 50MB
    client_max_body_size 50M;
}
```

### 5.2 try_files 详解

这是最容易让人困惑的配置，用例子解释：

```
用户访问: http://106.55.56.92/login

Nginx 的处理过程:
1. 先找 /var/www/.../dist/login 这个文件 → 不存在
2. 再找 /var/www/.../dist/login/ 这个目录 → 不存在
3. 最后返回 /var/www/.../dist/index.html → 存在！返回它

为什么这样设计？
因为 React 是单页应用（SPA），所有页面都是通过 JavaScript 渲染的。
真正的路由判断在 index.html 加载的 JavaScript 里完成。
所以不管访问什么路径，都要先返回 index.html。
```

### 5.3 Nginx 常用操作

```bash
# 测试配置文件是否正确（修改配置后必须先测试！）
nginx -t

# 重新加载配置（不中断服务）
systemctl reload nginx

# 重启 Nginx（会短暂中断）
systemctl restart nginx

# 查看状态
systemctl status nginx

# 查看访问日志（看谁在访问你的网站）
tail -f /var/log/nginx/access.log

# 查看错误日志（排查问题用）
tail -f /var/log/nginx/error.log
```

### 6.3 PM2 常用命令详解

#### 启动应用

```bash
# 基础启动
pm2 start app.js

# 指定应用名称（推荐！方便管理）
pm2 start app.js --name blog-api

# 监听文件变化，自动重启（开发时用）
pm2 start app.js --watch

# 集群模式：启动多个实例（生产环境用）
pm2 start app.js -i max  # max 表示 CPU 核心数
```

#### 查看状态

```bash
pm2 status
# 或
pm2 list
```

输出示例：
```
┌─────┬────────────┬─────────────┬─────────┬─────────┬──────────┬────────┬──────┬───────────┬──────────┐
│ id  │ name       │ namespace   │ version │ mode    │ pid      │ uptime │ ↺    │ status    │ cpu      │
├─────┼────────────┼─────────────┼─────────┼─────────┼──────────┼────────┼──────┼───────────┼──────────┤
│ 0   │ blog-api   │ default     │ 1.0.0   │ fork    │ 12345    │ 2D     │ 0    │ online    │ 0%       │
└─────┴────────────┴─────────────┴─────────┴─────────┴──────────┴────────┴──────┴───────────┴──────────┘
```

各列含义：
- **id**：PM2 内部编号
- **name**：应用名称
- **pid**：系统进程 ID
- **uptime**：运行时长（2D = 2天）
- **↺**：重启次数（如果频繁重启说明程序有问题）
- **status**：状态（online=运行中，stopped=已停止，errored=出错）
- **cpu**：CPU 占用率

#### 查看日志

```bash
# 查看所有日志
pm2 logs

# 查看指定应用的日志
pm2 logs blog-api

# 查看最近 100 行日志
pm2 logs blog-api --lines 100

# 实时监控日志（像看直播）
pm2 logs blog-api --follow
```

日志文件位置：
- 输出日志：`~/.pm2/logs/blog-api-out.log`
- 错误日志：`~/.pm2/logs/blog-api-error.log`

#### 重启/停止/删除

```bash
# 重启（更新代码后执行）
pm2 restart blog-api

# 停止（不删除，可以再启动）
pm2 stop blog-api

# 删除（从 PM2 列表中移除）
pm2 delete blog-api

# 重启所有应用
pm2 restart all
```

#### 保存和开机自启

```bash
# 保存当前进程列表
pm2 save

# 生成开机自启脚本
pm2 startup
# 会输出一条命令，复制执行它

# 取消开机自启
pm2 unstartup
```

### 6.4 PM2 监控面板

```bash
# 实时监控面板（很酷！）
pm2 monit
```

会显示一个实时更新的面板：

## 八、常见问题解答

### Q1：为什么不直接用 Node.js 处理所有请求？

**可以，但不推荐。原因：**

1. **性能**：Nginx 处理静态文件比 Node.js 快得多
2. **安全**：Nginx 作为屏障，后端不直接暴露
3. **稳定**：Nginx 非常稳定，几乎不会崩溃
4. **功能**：Nginx 提供 SSL、缓存、压缩等开箱即用的功能

### Q2：PM2 和直接用 `node app.js` 有什么区别？

| 方式 | 关闭终端后 | 程序崩溃后 | 日志管理 | 开机自启 |
|------|------------|------------|----------|----------|
| `node app.js` | 程序停止 | 程序停止 | 无 | 无 |
| `pm2 start` | 继续运行 | 自动重启 | 有 | 可配置 |

### Q3：Nginx 502 Bad Gateway 怎么办？

这意味着 Nginx 无法连接到后端。排查步骤：

```bash
# 1. 检查后端是否运行
pm2 status

# 2. 检查后端端口是否监听
netstat -tlnp | grep 3001

# 3. 检查 Nginx 配置的 proxy_pass 地址是否正确
cat /etc/nginx/conf.d/blog.conf | grep proxy_pass
```

### Q4：怎么查看网站访问量？

```bash
# 实时查看访问日志
tail -f /var/log/nginx/access.log

# 统计今天的访问量
cat /var/log/nginx/access.log | wc -l
```

### Q5：程序一直重启怎么办？

```bash
# 查看重启次数
pm2 status
# 如果 ↺ 列数字很大，说明程序频繁崩溃

# 查看错误日志找原因
pm2 logs blog-api --err --lines 50
```

常见原因：
- 数据库连接失败
- 端口被占用
- 代码有 bug

## 总结

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   Nginx = 前台接待员                                        │
│   - 接待所有访客（监听 80 端口）                            │
│   - 直接提供菜单（返回静态文件）                            │
│   - 把订单转给后厨（反向代理 API 请求）                     │
│                                                             │
│   PM2 = 厨师长                                              │
│   - 监督厨师工作（进程守护）                                │
│   - 厨师睡着了就叫醒他（崩溃自动重启）                      │
│   - 记录每道菜（日志管理）                                  │
│   - 餐厅开门厨师自动上班（开机自启）                        │
│                                                             │
│   它们配合：                                                │
│   用户 → Nginx → 静态文件 / API → PM2 管理的 Node.js       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

记住这个比喻，你就理解了 90% 的部署原理！
